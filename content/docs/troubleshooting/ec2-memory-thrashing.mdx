## 1. 핵심 용어 판단: livelock이 아니라 thrashing(스래싱)

### 결론

이 현상은 **livelock이 아니라 스래싱**(thrashing)이다.
처음에는 `livelock`이라는 용어로 접근했지만, 정확도 검토 결과 표준 개념인 **스래싱**이 맞다고 판단했다.

### 두 개념의 정의 비교

| 개념 | 정의 | CPU 상태 | 이번 현상과의 일치 여부 |
| --- | --- | --- | --- |
| **livelock** | 프로세스들이 **CPU에서 활발히 실행되면서도** 서로의 상태 변화에 반응하느라 유효한 진행을 못 하는 상태 (예: 두 스레드가 서로 양보만 반복) | **busy** (CPU 사용 중) | ❌ 불일치 |
| **deadlock** | 둘 이상의 프로세스가 서로가 쥔 자원을 영구히 기다려 아무도 진행 못 함 | idle (대기) | ❌ (자원 순환 대기 아님) |
| **thrashing** | 워킹셋이 물리 메모리를 초과해, 시스템이 유효한 작업 대신 페이지 회수·재적재(paging)에 시간을 다 쓰는 상태 | I/O wait (대기) | ✅ 일치 |

### 왜 livelock이 아닌가

- livelock의 핵심 특징은 **CPU가 바쁜데도 진행이 없다**는 것이다.
- 이번 현상은 정반대로, `vmstat`의 `wa%`(I/O wait)가 치솟고 프로세스 대부분이 **D state(uninterruptible sleep)** 에 묶여 **CPU를 거의 쓰지 않은 채** I/O 완료만 기다렸다.
- 즉 "활발히 도는데 진행이 없는" 것이 아니라 "멈춰서 I/O를 기다리는" 상태였다 → livelock의 정의와 맞지 않는다.

### 왜 thrashing이 맞는가

- Node.js 빌드 힙 급증으로 워킹셋이 가용 RAM(약 2.9GiB)을 넘어섰다.
- 커널이 페이지 회수와 재적재(refault)에 시간을 쏟느라 유효한 작업이 진행되지 못했다 → 스래싱의 교과서적 정의에 부합.
- 스래싱이 **디스크(EBS) I/O 포화**로 발현되면서 프로세스들이 D state에 쌓였다.

### "D-state thrashing"이라는 표현에 대해

한때 `D-state thrashing`이라는 복합어로 부르기도 했다. 서술적으로는 정확하지만 **교과서 표준 용어는 아니다**. 그래서

- **표준 용어 "스래싱"을 핵심 개념**으로 두고,
- **D state**는 "스래싱이 프로세스 단에서 관측되는 모습(메커니즘)"으로 배치

하는 방식으로 정리한다. 처음 접하는 사람도 표준 개념으로 이해할 수 있게 하기 위함이다.

## 2. 핵심 메커니즘: swap 유무가 결과를 가른 이유

swap이 있고 없고가 결과를 가른 이유를 표준 리눅스 메모리 회수 관점에서 정리한다.

### 리눅스 메모리 회수 대상은 두 종류

| 종류 | 예시 | 회수 방법 |
| --- | --- | --- |
| **익명 페이지(anonymous page)** | 힙, 스택 — 파일에 대응되지 않는 메모리 | **swap이 있어야만** 디스크로 내보낼 수 있음 |
| **파일 기반 페이지(file-backed page)** | 페이지 캐시 — 실행 중 프로세스의 코드/라이브러리 포함 | 파일에서 다시 읽으면 되므로 그냥 버릴 수 있음 |

### swap이 없을 때 (실제 장애 / Phase 1)

1. 커널은 익명 페이지를 내보낼 곳이 없음 → **파일 기반 페이지만 회수 가능**.
2. 회수 대상 안에 **지금 실행 중인 프로세스의 코드 페이지**까지 포함됨.
3. 코드 페이지를 회수 → 즉시 다시 필요 → EBS에서 재적재(refault) → 또 회수 → 또 읽기 … **재페치 폭주**.
4. 이 폭주가 EBS IOPS를 포화시킴 = 스래싱의 실체.

### swap이 있을 때 (Phase 2)

1. 커널이 **익명 페이지를 swap으로 내보낼 수 있음** → 파일 캐시(코드 페이지) 회수 압박이 풀림.
2. 코드 페이지가 메모리에 상주 → 재페치 악순환이 끊김.
3. 실제 swap 사용량은 **208MB에 불과**했지만, "익명 페이지를 내보낼 수 있느냐 없느냐" 자체가 스래싱 발생 여부를 갈랐다.
   → 작은 양이라도 배출구의 존재가 임계 동작을 바꿨다는 것이 핵심.

### "왜 OOM Kill이 안 떴는가"도 같은 메커니즘으로 설명됨

- OOM Killer는 커널이 **회수 가능한 메모리가 더 없다**고 판단할 때 발동한다.
- swap이 없어도 **회수 가능한 페이지 캐시는 계속 남아 있으므로**, 커널은 "아직 회수할 게 있다"고 보고 프로세스를 죽이는 대신 회수·재페치를 무한 반복한다.
- 결과: **OOM 로그는 안 남고, 시스템만 스래싱으로 멈춘다.**
- 이것이 "가설 2(OOM) 탈락" → "진짜 원인(스래싱)"을 잇는 논리적 다리다.

> 참고: 이 "저메모리 + no-swap 무응답" 실패 모드는 리눅스에서 잘 알려진 현상이다.
> PSI(Pressure Stall Information), `le9` 패치 논의, Artem S. Tashkinov의 커널 버그질라 리포트("Linux is dying under low memory") 등에서 동일한 메커니즘이 다뤄진다.

## 3. 근거 체인 (관찰 → 가설 → 검증)

핵심 주장을 뒷받침하는 증거를 한눈에.

### 배제한 가설

| 가설 | 확인 방법 | 결과 |
| --- | --- | --- |
| CPU 크레딧 소진 | CloudWatch `CPUCreditBalance` | 576 (거의 만충) → 탈락 |
| OOM Kill | `dmesg`, `journalctl -k` grep oom | 로그 없음 → 탈락 (단, 이게 스래싱의 단서) |

### 스래싱 가설을 지지하는 증거

| 증거 | 정상 | 장애/재현 | 해석 |
| --- | --- | --- | --- |
| `VolumeQueueLength` (실제 장애) | ≈ 0 | 1.68 | EBS 앞에 I/O 큐 쌓임 — **신호**이지 단독 확정 근거는 아님 |
| `VolumeQueueLength` (Phase 1 재현) | ≈ 0 | **27.62** | `--no-cache`로 더 가혹 → 큐 폭증, 가설 강하게 지지 |
| SSH 응답 | 정상 | 불가 | 단일 프로세스가 아닌 사용자 공간 전반 정지 |
| `wa%` (Phase 2) | 0% | 30% 순간 후 안정 | swap이 I/O wait 폭주를 막음 |
| swap 사용량 (Phase 2) | — | 208MB | 익명 페이지 배출로 악순환 차단 |

### 검증 설계의 핵심

- **원인 확정과 해법 검증을 분리**했다.
  - Phase 1: swap OFF → 장애 재현 (원인 확정)
  - Phase 2: swap ON + **더 불리한 캐시 조건(2.9GiB > 2.3GiB)** → 빌드 완료 (해법 검증)
- 해법 검증을 더 가혹한 조건에서 했기 때문에, "운이 좋아서 통과한 것 아니냐"는 반론을 차단한다.

## 4. 간헐성·릴리즈 분석의 근거

### 간헐적이었던 이유 = 두 조건의 중첩

1. **빌드 피크 힙의 점진적 증가** (릴리즈마다 누적)
2. **빌드 시점 페이지 캐시 누적량의 변동** (재부팅 직후 적음 → 운영하며 증가)

두 조건이 동시에 임계점을 넘을 때만 스래싱 발현 → 매번이 아니라 간헐적.

### "왜 하필 그 릴리즈" — 코드 베이스 비교 근거

(버전은 상대 표기. `vN` = 최초 hang 버전)

| 버전 | `schema.ts` | TS 파일 수 | 비고 |
| --- | --- | --- | --- |
| vN-9 ~ vN-4 | 약 43,500줄 | 약 1,200개 | 6버전 동결 |
| vN-3 | +약 150줄 | +11개 | 성장 재개 |
| vN-2 | +약 220줄 | +1개 | |
| vN-1 | 변동 없음 | 변동 없음 | 정상 |
| **vN** | **+약 280줄** | **+12개** | 첫 hang |

- `schema.ts`는 OpenAPI 자동 생성 파일이라 줄지 않고 누적만 됨.
- TS 컴파일러가 이 파일의 AST 전체를 메모리에 올린 뒤 참조 파일(1,000개 이상)을 타입 추론 → 참조 증가 시 컴파일 메모리가 늘어남.

#### ⚠️ 정확도 주의 (과장 완화 지점)

- 컴파일 메모리가 **"승수 효과(multiplicative)"** 로 증가한다고 단언하면 **과장**이다.
- TS 타입 추론이 병리적 케이스에서 초선형(superlinear)일 수는 있으나, 구체적 측정 없이 "승수 효과"는 근거가 부족하다.
- 따라서 **"선형이 아니라 더 가파르게 증가"** 정도로 표현하는 게 안전하다.

## 5. swap 설정(swappiness)의 미묘한 지점

- 적용: `vm.swappiness=10` (기본 60에서 낮춤).
- 목적: 평상시 서비스 응답 지연을 피하려고 swap을 **최후의 완충 장치**로만 쓰려는 운영 판단.

### swappiness를 낮추는 실무적 효과 (선제적 swap 억제)

- swappiness는 **swap 사용량의 상한/throttle이 아니라**, 메모리 회수 시 "익명 페이지를 swap으로 내보낼까 vs 파일 캐시를 버릴까"의 **선호 비율**이다.
- 기본값 60에서는 RAM이 위급하지 않아도 커널이 잘 안 쓰는 익명 페이지를 **선제적으로** swap에 빼 두는 경향이 있다(그 자리를 페이지 캐시로 활용). 그 결과 평상시에도 swap 사용량이 슬금슬금 차오르고, 해당 페이지를 다시 접근할 때 디스크 읽기로 지연이 생긴다.
- swappiness=10은 이 **불필요한 선제적 swap을 억제**한다 → 결과적으로 "swap을 켠 뒤 평상시 swap 사용량이 확 증가하는 현상"을 막아준다.
- 단, 이것은 **하드 상한이 아니라 선호 설정**이므로, 진짜 메모리 압박(이번 스래싱 같은 임계 상황)에서는 swappiness가 낮아도 swap이 늘어난다 — 그게 의도된 안전장치 동작이다.

### ⚠️ 솔직하게 짚어둘 긴장 관계

- swappiness를 **낮추면** 커널은 (swap보다) **파일 캐시 회수를 선호**한다.
- 그런데 이번 스래싱의 원인이 바로 **파일 캐시 회수·재페치**였다 → 표면적으로는 우리가 노린 메커니즘과 상충하는 방향.
- 다만 **메모리 압박이 심해지면 swappiness가 낮아도 커널은 익명 페이지를 swap으로 내보낸다.** 우리가 노린 "피크 흡수" 효과는 임계 상황에서 그대로 작동한다.

## 부록: 핵심 개념 쉽게 풀어보기

위 개념들을 비유 중심으로 다시 정리한다. 처음 접하는 사람도 따라올 수 있게 하는 게 목적.

### 큰 그림 비유: 사무실, 책상, 창고

| 비유 | 실제 | 비고 |
| --- | --- | --- |
| **책상** | 물리 메모리(RAM) | 빠르게 펼쳐놓고 작업하는 공간. 한정적. |
| **창고/서랍** | 디스크(EBS) | 느리지만 많이 보관. |
| **창고 안 "익명 서류 전용 보관함"** | swap | 디스크 위에 있어 **느림**. "임시 책상"이 아니라 비상 보관함. |
| **책상 관리자** | 커널(kernel) | 공간을 배분하고, 안 쓰는 서류를 치우고, 필요하면 다시 올려줌. **직접 일하지는 않음.** |
| **일하는 워커** | 프로그램/프로세스 (`node`, `webpack`, `tsc`) | 서류 내용을 **읽고 참고해 실제 작업**을 함. |

핵심 구분: **커널은 공간 관리자**, **프로그램은 그 공간을 쓰는 사용자**다. 서류를 "올리고 치우는" 건 커널, 서류 내용을 "보고 일하는" 건 프로그램.
그리고 **swap은 작업 공간을 넓혀주는 게 아니라**(느린 디스크라서), 책상이 터지지 않게 해주는 비상 보관함이다. → "swap은 성능 개선책이 아니라 안전장치"인 이유.

### 워킹셋 / 회수 / 재적재

- **워킹셋(working set)** — 프로그램이 할당한 전체 메모리가 아니라, **지금 이 순간 활발히 들여다보는 부분**. = 책상에 펼쳐놔야 하는 서류.
  - 워킹셋 ≤ RAM → 필요한 서류가 다 책상 위. 빠름.
  - 워킹셋 > RAM → 다 못 올림 → 서랍에 넣었다 뺐다 반복. 느림 = 스래싱의 시작.
- **회수(reclaim)** — 책상이 꽉 차서, 안 쓰는 서류를 치워 공간을 비우는 행위.
- **재적재(refault)** — 방금 치운 서류가 곧바로 또 필요해져 창고에서 다시 꺼내 올리는 것. 디스크에서 다시 읽으니 느림.
- 워킹셋 > RAM이면 "치웠다 → 또 필요 → 다시 읽고 → 또 치우고"가 반복됨 = **스래싱**.

### `vmstat`과 `wa%`

- **`vmstat 2`** — 2초마다 시스템(메모리·CPU·디스크) 상태를 한 줄씩 출력. 장애 구간 변화를 시간순으로 관찰 가능.
- CPU 컬럼: `us`(사용자 코드) / `sy`(커널) / `id`(놀고 있음) / **`wa`(I/O wait)**.
- **`wa%` = CPU가 디스크 I/O 끝나길 기다리느라 아무 일도 못 하고 노는 비율.**
  - 높다 = CPU 성능 문제가 아니라 **디스크가 못 따라가서 시스템이 묶여 있다**는 신호.
  - Phase 2: 30% 순간 스파이크 후 안정화(swap이 디스크 대기를 흡수).
  - Phase 1: 100% 가깝게 치솟았을 것으로 추정(SSH 차단으로 직접 측정 불가).

### 프로세스 상태와 D state

`ps`/`top`의 `STAT` 컬럼 한 글자:

| 코드 | 의미 |
| --- | --- |
| `R` | 실행 중/실행 대기 |
| `S` | 대기 중, **신호로 깨울 수 있음** (interruptible) |
| `D` | 대기 중, **신호로도 못 깨움** (uninterruptible) |
| `Z` | 종료됐지만 부모가 미수거 (zombie) |

- **`D` state** = 저수준 I/O(디스크 등)가 **물리적으로 끝날 때까지** 커널이 재워둔 상태. `kill -9`도 안 통함(I/O 중간에 깨우면 데이터 손상 위험 → 일부러 못 끼어들게 막음).
- 평소엔 잠깐 들어갔다 빠져나오지만, **디스크가 포화되면** I/O가 안 끝나 프로세스들이 `D`에 쌓임.
- **sshd까지 `D`에 빠지면 SSH도 무응답** → "SSH까지 막힘"이 단일 프로세스 문제가 아니라고 본 근거.

### Node.js 힙이 빌드 때 급증하는 이유

힙(heap) = V8이 객체·변수를 담는 메모리. `next build`가 힙을 많이 먹는 이유:

1. **TypeScript 타입 체크** — `schema.ts`(4만 줄 이상)의 AST를 메모리에 올리고, 참조하는 1,000개 이상 파일 타입 추론.
2. **webpack 번들링** — 전체 모듈을 메모리에 올려 의존성 그래프 구성·번들·최적화.
3. **압축/소스맵** — 변환 중간물을 메모리에 들고 있음.
4. **워커 스레드** — 작업을 나눈 워커들이 각자 V8 힙 보유.

→ 빌드 특정 순간 힙이 2GB+로 치솟음. 릴리즈마다 코드가 늘며 이 피크가 높아져 어느 순간 가용 RAM(2.9GiB)을 넘긴 게 "왜 하필 그 릴리즈".

### 익명 페이지 vs 파일 기반 페이지 (제일 중요)

메모리는 **페이지**(보통 4KB) 단위로 관리됨. 페이지는 "비우는 방법"에 따라 두 종류로 나뉜다.

| | **파일 기반 페이지(file-backed)** | **익명 페이지(anonymous)** |
| --- | --- | --- |
| 정체 | 디스크 파일에서 읽어온 사본 | 실행 중 계산해 만든 데이터 |
| 예시 | `node` 바이너리 코드, 라이브러리(`.so`), 읽어들인 소스 파일, Docker 레이어 | V8 힙(객체·문자열), TS AST/타입 정보, webpack 모듈 그래프, 워커 스택 |
| 원본 위치 | **디스크 파일에 있음** | **RAM에만 있음** (원본 없음) |
| 그냥 버려도 되나 | ✅ (필요하면 파일에서 다시 읽음) | ❌ (버리면 영영 사라짐 → 프로그램 깨짐) |

### swap이 회수에 미치는 영향 (정확한 표현)

**페이지를 비우려면 데이터가 사라지지 않을 보장이 필요하다.** (a) 이미 다른 곳에 있거나, (b) 비우기 전 어딘가에 써둘 수 있어야 함.

- **파일 기반** → 원본이 디스크에 이미 있음 → **swap과 무관하게 항상 회수 가능.**
- **익명** → 원본이 RAM에만 있음 → 비우려면 어딘가 써둬야 하는데, **그 유일한 보관처가 swap.**
  - swap 있음 → swap에 써두고 비울 수 있음 → **회수 대상이 됨.**
  - swap 없음 → 써둘 곳 없음 → **회수 대상에서 빠짐.**

| | swap 없을 때 회수 | swap 있을 때 회수 |
| --- | --- | --- |
| 파일 기반 페이지 | ✅ 가능 (원래부터) | ✅ 가능 |
| 익명 페이지 | ❌ 불가 | ✅ 가능 (swap에 써두고) |

→ **swap의 진짜 역할은 "익명 페이지를 회수 가능 목록에 추가"하는 것 하나.** (파일 기반은 swap과 무관하게 늘 회수 가능.)
흔히 "swap 있으면 둘 다 회수 대상"이라고 표현하지만, 정확히는 "익명 페이지가 비로소 회수 대상에 합류"하는 것이다.

### 악순환으로 연결

1. swap 없음 → 익명 페이지(빌드가 만든 거대한 힙·AST) 못 비움 → **파일 기반 페이지만** 비울 수 있음.
2. 그 안엔 **지금 실행 중인 코드 페이지**도 포함됨.
3. 메모리가 너무 부족 → 코드 페이지까지 비움 → 곧바로 또 필요 → 디스크에서 재적재(refault).
4. 읽어오자마자 또 부족 → 또 비움 → 또 읽음 → **무한 반복** = EBS I/O 폭주 = 스래싱.
5. swap 있으면 익명 페이지를 창고(swap)로 보내, 코드 페이지를 건드릴 일이 줄어 악순환이 끊김.
   → 실제 swap 사용량 208MB로도 효과가 컸던 이유: **"익명 페이지를 내보낼 수 있느냐" 자체가 악순환의 스위치**였기 때문.
