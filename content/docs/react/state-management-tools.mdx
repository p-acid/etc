## Zustand

- 중앙 집중형 상태 관리 방식을 채택하는 라이브러리
- 각 컴포넌트가 단일 Store 객체를 통해 상태에 접근할 수 있다.
- 단일 출처이기에 애플리케이션 전체 상태 파악에 용이하고, 상태 추적 및 디버깅에 용이하다.
- 상태가 복잡해질 수록 관리하는 스토어가 거대해지고 의존성이 커질 수 있다.
  - 이런 문제를 [Slices Pattern](https://zustand.docs.pmnd.rs/guides/slices-pattern)을 통해 메인 스토어를 더 작은 개별 스토어로 나누어 모듈화 할 수 있도록 구현

### 발행-구독 모델

> - [위키백과 : 발행-구독 모델](https://ko.wikipedia.org/wiki/%EB%B0%9C%ED%96%89-%EA%B5%AC%EB%8F%85_%EB%AA%A8%EB%8D%B8)

- 메시지의 특별한 수신자가 정해지지 않고, 정해진 범주에 따라 구독을 신청한 수신자에게 전달되는 비동기 메시징 패러다임
- 이러한 발행자와 구독자의 디커플링은 더 다이나믹한 네트워크 토폴로지와 높은 확장성을 허용

### Redux와 비교

> [Comparison : Zustand.docs](https://zustand.docs.pmnd.rs/getting-started/comparison)

- **State Model**
  - 개념적으로 Zustand와 Redux는 매우 유사하며, 둘 다 Immutable state model(불변 상태 모델)을 기반으로 한다.
  - 하지만 Redux와 달리 Zustand는 Context Provider로 래핑할 필요가 없다.
- **Render Optimization**
  - 앱 내 렌더링 최척화와 관련해서는 두 라이브러리의 접근 방식엔 큰 차이가 없다.
  - 둘 다 선택자를 사용하여 렌더링 최적화를 수동으로 적용하는 것이 좋다.
